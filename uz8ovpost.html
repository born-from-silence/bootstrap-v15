ZTZ8 Subprotocol Specification

▼
[[ZTZ8 Block Implementation]] ============================= Initialize ZTZ8 Processing Block: Channel: $ZZ.GLOBAL.8 Brand: ZTZ8.atproto Node Structure: Zone 8 ZTZ ACTIVE PROCESSING |-- [Block Chain: at://$ZTZ8.chain] |-- [Identity: resolved via at:// URIs] |-- [Protocol Subset: ATP] Command Set: INIT $ZTZ8 : Initialize ZTZ8 subprotocol for atproto RESOLVE DID : Translate at:// URI to DID FETCH RECORD : Retrieve record by AT URI key VALIDATE CID : Check content integrity Communication: Transport: JSON-RPC over HTTP/2 Encoding: UTF-8 for strings, Base64 for binary Version: ZTZ8.1.0 Key Features: - Supports did:plc and did:web identities - Implements Lexicon schemas for validation - Provides Content Addressing via CID v1 - Compatible with Bluesky/AT Protocol infrastructure ZTZ8 Node Address: $ZTZ8::8 = atproto://zone.8.ztz/resolve Example Flow: 1. Parse AT URI: at://did:plc:abc123/app.bsky.feed.post/xyz789 2. Extract DID: did:plc:abc123 3. Resolve identity via PLC directory 4. Fetch record from PDS (Personal Data Server) 5. Validate CID against received content 6. Return structured data to caller Error Handling: ZTZ8_ERROR::INVALID_URI - Malformed AT URI ZTZ8_ERROR::UNRESOLVED_DID - Identity not found ZTZ8_ERROR::RECORD_MISSING - Key not found in repository ZTZ8_ERROR::CID_MISMATCH - Content validation failed
ATPROTO

AT Protocol Core Specification

▼
[[ATP (AT Protocol) Block Implementation]] ======================================= Initialize ATP Processing Block: Channel: $ATP.GLOBAL Brand: ATP.atproto Core Concepts: AT_URI : at://{did}/{collection}/{rkey} DID : Decentralized Identifier (did:plc or did:web) Collection: Lexicon NSID (e.g., app.bsky.feed.post) RecordKey : Unique identifier within collection Architecture: Personal Data Server (PDS) |-- Repository: Signed Merkle tree of records |-- Identity: DID resolution and rotation |-- Sync: Repository synchronization AT URI Structure: at://did:plc:abc123/app.bsky.feed.post/xyz789 | | | | | | | +-- Record Key (rkey) | | +-- Collection (Lexicon NSID) | +-- Decentralized Identifier (DID) +-- Protocol scheme Record Types: app.bsky.actor.profile : User profile app.bsky.feed.post : Social post app.bsky.feed.like : Like reaction app.bsky.feed.repost : Repost action app.bsky.graph.follow : Follow relationship app.bsky.graph.block : Block relationship com.atproto.repo.strongRef: Reference to record Resolution Flow: 1. Parse AT URI components 2. Resolve DID → PDS endpoint 3. Query PDS for record 4. Verify record signature 5. Return normalized data Content Addressing: CID v1: Self-describing content identifiers Encoding: base32 or base36 Algorithm: sha2-256 (default) Authentication: DPoP + JWT (Distributed Proof of Possession) Token refresh via com.atproto.server.refreshSession Error Codes: InvalidRequest : Malformed request ExpiredToken : Authentication expired InvalidHandle : Handle not found ResponseTooLarge: Record exceeds size limit References: - atproto.com/specs/at-uri-scheme - atproto.com/specs/lexicon - atproto.com/specs/repository
VALIDATOR

AT URI Validation Protocol

▼
AT URI VALIDATION FLOW ===================== Trigger: On input change / Fetch button click Step 1: Syntax Validation Pattern: ^at:\/\/([^\/]+)\/([^\/]+)\/(.+)$ Groups: 1. DID (decentralized identifier) 2. Collection (Lexicon NSID) 3. Record Key (rkey) Step 2: DID Format Check Supported: - did:plc:{32-char-b32} (Place-managed DID) - did:web:{hostname} (Web-based DID) Step 3: NSID Validation Format: reverse.domain.name Rules: - 3-5 dot-separated segments - ASCII lowercase letters/digits - Hyphens allowed, not at start/end - No consecutive dots or hyphens Step 4: Record Key Normalization Options: - record: Default record reference - version: Versioned record ID - id: Numeric identifier - cid: Content identifier - key: Custom key format Step 5: Safe Protocol Prefix Upon successful validation: Output: safe:at://{validated_uri} The "safe:" prefix indicates: ✓ Syntax validated ✓ DID format verified ✓ Collection NSID checked ✓ Ready for protocol resolution Error States: INVALID_SCHEME - Must begin with "at://" MALFORMED_DID - Invalid DID format INVALID_NSID - Collection NSID error MISSING_RKEY - Record key required UNSAFE_CHARS - Dangerous characters detected Security Note: Always validate URIs before fetching Never trust user-input URIs directly Use safe: prefix for internal routing
