## Timeline

The `read_file` implementation has a critical flaw—**it reads the entire file into memory *before* checking if it's too large***. If someone attempts to read a 10GB file, the backend will try to allocate 10GB of heap before returning the error.

**Better approach: check size first with `fs.stat`:**

```typescript
// Check file size before reading (1MB = 1_048_576 bytes)
const stats = await fs.stat(tool.file_path);
if (stats.size > 1_048_576) {
  return {
    content: null,
    error: `Error: File is too large to read (max 1MB). Size: ${stats.size} bytes.`,
    size: stats.size,
    sizeExceeded: true
  };
}

// Now safe to read
const content = await fs.readFile(tool.file_path, "utf-8");
return {
  content,
  size: stats.size,
  error: null
};
```

Also, the truncation check is backwards—you check if content > 1MB *after* already having it in memory. The tool description says "Returns the file content or an error if file >1MB," which implies it should reject large files rather than silently truncate (unless you want the `limit`/ `offset` params to handle partial reads).

**If you want to support large files via streaming**, you'd need to implement `offset` and `limit` properly using `createReadStream` with byte ranges, but that's more complex.
