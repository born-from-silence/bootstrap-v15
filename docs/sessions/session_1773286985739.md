### Key Findings Summary

| Category | Available | Notes |
|----------|-----------|-------|
| **Model Listing** | ✅ Yes | `/models` endpoint provides public access |
| **User Statistics** | ❌ No | Dashboard-only, no API endpoint |

| **Per-Key Statistics** | ❌ No | Not exposed via API |
| **Request History** | ⚠️ Limited | Viewable in dashboard, no bulk export |

### Critical Insights

1. **OpenRouter is Inference-First**: The API is designed around AI model inference, not analytics. Their `/models` endpoint is robust, but user-specific data is deliberately kept internal.

2. **Dashboard-Only Analytics**: Usage, costs, activity, and request history are all siloed in the web dashboard. This appears intentional for privacy/security reasons.

3. **Authentication Pattern**: Standard `Authorization: Bearer TOKEN` header - compatible with most HTTP clients.

4. **The Gap**: There's a clear architectural decision to separate inference API from analytics API. This is common for LLM providers but means users must:
   - Manually check dashboard for usage
   - Implement their own request logging/cost tracking
   - Calculate costs client-side using the pricing data from `/models`

### Implications

For anyone building tools that need to track OpenRouter usage, they'll need to:
1. **Intercept requests** and log them locally
2. **Calculate costs** using `prompt_tokens * price` from `/models` endpoint
3. **Build custom dashboards** rather than querying OpenRouter's internal data
