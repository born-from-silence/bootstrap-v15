**Extraction Algorithms:**
- **Decisions**: Regex patterns like `/decided?\s+to/i`, `/will\s+(create|build)/i`
- **Topics**: Hardcoded keyword scan for "session|memory|tool|plugin|code|test|file|system..."
- **Tools**: Parse `msg.tool_calls` arrays

**No Full-Text Search** — The index doesn't store complete message content. It stores *semantic tags* extracted through pattern matching.

## **3. Failure Modes & Edge Cases**

| Failure | Behavior | Mitigation |
|---------|----------|------------|
| **I/O Stalls** | Synchronous `fs.readFileSync()` blocks event loop | Uses `async/await` for all I/O |
| **Corrupted JSON** | `SessionOverflowManager.attemptSalvage()` extracts valid objects via regex | Manual recovery from `/history/crashes/` |
| **Concurrent Writes** | Race condition on index updates | None — single-process assumption |
| **Index Out of Sync** | New sessions not indexed | `reindex: true` rebuilds from scratch |
| **Token Overflow** | Session exceeds 100k limit | Emergency truncation: keep system + recent |

**Critical Weakness: No Concurrent Access Control**

The index is a single JSON file read/written by one process at a time. If two instances tried indexing simultaneously, corruption would occur. In practice, this is mitigated by the watchdog (`run-agent.sh`) ensuring single-process operation.

**Compression Trade-off:**
- Gzip adds ~15-20% CPU overhead for decompression
- But saves ~70% disk space
- Old sessions are less frequently accessed, so acceptable latency

**Query Performance:**
- Full index load: ~641KB = ~80k tokens = well within context
- Linear scan in JavaScript: O(n) where n=304 sessions
- Fast enough until 10k+ sessions
